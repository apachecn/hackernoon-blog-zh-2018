成为优秀程序员的139个特质

原文：https://medium.com/hackernoon/139-traits-that-make-a-better-programmer-4d0d916b9f78?source=collection_archive---------5-----------------------#2018-07-23

你应该记住的事情。

让我们面对现实吧，每个开发人员都想改进和变得更好。大多数开发人员问的最大问题是我将如何做。在编程时，这里有一些你应该记住的事情。

代码表示

  • 随着任何关于代码布局的讨论越来越多，它陷入毫无结果的争论的可能性越来越大。你知道制表符和空格，新行或同一行的括号位置。
  • 好的代码格式不是你认为看起来最漂亮的。一个好的格式是使代码更容易扫描和阅读的格式。
  • 良好的代码表示揭示了代码的意图。这不是一种艺术努力。
  • 我们需要良好的表现，以避免代码错误。不是为了让我们创造出漂亮的ASCII艺术。
  • 我们写代码是为了让计算机执行，但却是为了让人类阅读。
  • 如果有人想写一种清晰的风格，首先让他的思想清晰。
  • 选择一种布局样式并一致使用，或者采用编码标准或样式指南。
  • 如果您处理的文件不符合项目其余部分的布局约定，请遵循该文件中的布局约定。

命名

  • 一个好的名字是描述性的、正确的、符合习惯的。
  • 你只有确切地知道它是什么，才能很好地说出它的名字。在给变量起神秘的名字之前，确保你知道它是什么。
  • 避免冗余。这表明你没有考虑清楚你正在建造的东西的名字。无论是类、函数还是变量
class ItemList {public int numberOfItems () {…}}
// numberOfItems is unnecessarily long, repeating the word Item
// it could be renamed to size or even length
  • 使用你的语言中最常用的大写约定。就像在C#中，方法是大写的Console.WriteLine("Hello World")
  • 确保您的姓名准确无误，避免拼写错误。
  • 永远不要同时改变表现和行为。使它们成为单独的受版本控制的更改。
  • 随着经验的积累，调整你的演讲风格。作为初学者，你并不总是从良好的代码表示开始，你更关心的是程序是否工作。

少写代码

  • 写很多代码并不意味着你已经写了很多软件。这可能只是意味着你写了很多错误。
  • 更多的代码意味着有更多的东西需要阅读和理解——这使得我们的程序更难理解。
  • 你拥有的代码越多，隐藏bug的地方就越多，追踪它们可能并不容易。

避免不必要的代码

无意义代码的常见类别是不必要的条件语句和重言式逻辑结构的使用。浮躁的逻辑是浮躁思想的标志。

// example
if (expresion) {
  return true;
} else {
  return false;
}
// which can be refactored to
return expresion;
  • 清晰简洁地表达代码。避免不必要的长篇大论。它们不会给你的代码增加任何价值。

复制

  • 不要复制代码段。将它们分解成一个公共函数。使用参数来表示任何差异。
  • 如果你发现重复，删除它。

死代码

  • 死代码是永远不会运行的代码，永远无法到达。
  • 告诉你的代码，要么生活，要么迷失。

死代码的其他表现还有:

  • 从未被调用的函数
  • 已写入但从未读取的变量
  • 传递给从未使用过的内部方法的参数
  • 从未使用过的枚举、结构、类或接口

评论

  • 好的代码不需要大量的注释来支撑它，或者解释它是如何工作的。
  • 确保每个注释都增加了代码的价值。代码本身说明了内容和方式。评论应该解释原因——但只有在还不清楚的情况下。
  • 不要通过注释掉代码来移除它。它会让读者感到困惑，并妨碍阅读。
  • 每天，让你的代码比以前更好一点。删除你发现的冗余和重复。

通过删除代码来改进代码

  • 您可以通过添加新代码来改进系统。您还可以通过删除代码来改进系统。

开发过程中死代码是如何冒出来的

  • 从应用程序的用户界面中删除了一些特性，但是保留了后端支持代码。
  • 向导生成的UI代码插入了一些经常不使用的钩子。
  • 尽可能删除死代码。它会妨碍你，让你慢下来。
  • 删除你在未来可能需要的代码是安全的。您总是可以从版本控制中取回它。
  • 代码清理应该总是在单独的功能变更提交中进行。
  • 即使是最好的代码库也会出现死代码。

警惕过去

当你写它的时候，你认为它是完美的，但是当你审视你的旧代码时，你会不可避免地发现各种各样的代码漏洞。

  • 回顾你的旧代码会让你知道你的编码技能的改进(或其他)。

如何使用现有的代码库

进入任何大型现有代码库都很困难。你必须迅速:

  • 发现从哪里开始查看代码
  • 找出代码的每一部分是做什么的，以及它是如何实现的
  • 衡量代码的质量
  • 找出如何在系统中导航
  • 理解编码习惯用法，这样你的改变将会令人同情
  • 找到任何功能的可能位置(以及由此导致的错误)
  • 编写代码的最佳途径是由已经熟悉地形的人带领。不要害怕寻求帮助！
  • 学习代码的最好方法是修改它。然后从你的错误中学习。
  • 许多程序员不愿意花力气去阅读和理解现有的代码，而宁愿说“它很难看”并重写它。
  • 做好遇到糟糕代码的准备。在你的工具箱里装满锋利的工具来对付它。
  • 一些肮脏的代码仅仅是由能力较差的程序员编写的。或者是一个有能力的程序员在糟糕的一天。
  • 当你遇到“糟糕”的代码时，不要有反感的感觉。相反，寻找切实改善它的方法。
  • 你不能期望任何代码都是完美的，即使是你自己的代码。
  • 遵循童子军规则。每当你接触一些代码时，让它比你发现它时更好。
  • 缓慢而小心地修改代码。一次做一个改变。

处理错误

  • 不要忽略代码中可能存在的错误。不要把处理错误推迟到“以后”(你不会有时间去做)。
  • 严格使用异常。理解你的语言的习惯用法和有效使用异常的要求。
  • 程序员必须意识到编程错误。必须让用户意识到使用错误。
  • 仅仅记录错误(在某个地方)是不够的，并希望一个勤奋的操作员有一天会注意到一个错误并对此采取措施。

为意外做好准备

  • 在每一步，考虑所有可能发生的不寻常的事情，不管你认为它们发生的可能性有多大。
  • 始终考虑可以恢复的错误，并编写适当的恢复代码。
  • 确保您的错误处理是惯用的，并使用适当的语言机制。
  • 在编写代码时，考虑所有潜在的代码路径。不要计划以后处理“不寻常”的情况:你会忘记，你的代码会有错误。

捕虫

  • 程序员写代码。程序员不是完美的。程序员的代码并不完美。因此，第一次并不完美。所以我们有虫子。
  • 我们应该始终采用合理的工程技术，最大限度地减少令人不快的意外发生的可能性。
  • 调试比一开始写代码要难两倍。因此，如果你尽可能聪明地编写代码，从定义上来说，你没有足够的聪明去调试它。
  • 将bug减少到最简单的一套 繁殖 步骤的可能。
  • 确保你只关注一个问题。
  • 断言和日志(even the humble console.log and nodejs assert)是强大的调试工具。经常使用它们。
  • 二进制chop问题空间，以获得更快的结果。
  • 当你开发你的软件时，花时间写一套单元测试。
  • 未经测试的代码是滋生bug的温床。测试是你的漂白剂。
  • 学习如何使用好你的调试器。然后在适当的时候使用它。
  • 尽快修复错误。不要让它们堆积起来，直到你陷入代码粪坑。
  • 调试并不容易。但这是我们自己的错。我们写了bug。

总是测试你的代码

单元测试专门测试最小的功能“单元”,以确保它们都能正确运行。这种隔离特别意味着单元测试不会涉及任何外部访问:不会运行任何数据库、网络或文件系统操作。

  • 质量是免费的，但只对那些愿意花大价钱购买的人开放。
  • 为了改进我们的软件开发，我们需要快速的反馈，以便在问题一出现就了解它们。好的测试策略提供短的反馈循环。
  • 在编写被测代码的同时编写测试。不要推迟测试写作，否则你的测试将不会有效。
  • 所有测试都应该作为持续集成工具链的一部分在您的构建服务器上运行。
  • 测试应用程序中重要的东西。
  • 全局变量和单例对象是可靠测试的梦魇。你不能轻易地测试一个有隐藏依赖的单元。
  • 分解你的代码使其“可测试”会导致更好的代码设计。
  • 程序测试可以用来显示缺陷的存在，但永远不能显示缺陷的不存在。

如何应对复杂性

  • 简单是一种伟大的美德，但它需要努力工作才能实现，需要教育才能欣赏。更糟糕的是:复杂性卖得更好。
  • 复杂性通常是偶然的，很少是有人故意添加的。
  • 推迟设计决策，直到你不得不采取它们。当您还不知道需求时，不要做架构决策。不要猜测。

熟能生巧

  • 程序员需要良好的品味和审美意识来编写出色的代码。
  • 任何聪明的傻瓜都能把事情变得更大、更复杂、更暴力。朝着相反的方向前进需要一点天赋和很大的勇气。
  • 好的软件开发不是牛仔式的编码，扔下你能想到的第一个代码。这是一个深思熟虑的，精确的努力。
  • 优秀的程序员工作起来很谦虚。他们承认他们并不知道全部。
  • 好的代码和好的编码者都是从用正确的方式写正确的东西的渴望中诞生的。

成为一名团队成员

  • 编程团队有一套规则。这些规则定义了我们做什么以及如何做。但是它们也描述了一种编码文化。
  • 不要依赖模糊的不成文的团队“规则”将隐含的规则明确化，并控制你的编码文化。
  • 将错误修复应用于根本原因，而不是症状出现的地方。贴膏药式的症状修复不会导致简单的代码。
  • 在代码中避免隐含的假设。
  • 只写需要的代码。任何多余的东西都是会成为负担的复杂性。
  • 停下来想一想。不要写愚蠢的代码。
  • 承认你的错误和糟糕的编码决策。向他们学习。
  • 要有动脑子的勇气。感觉被授权去评论代码并决定如何改进它。
  • 了解是什么让软件变得容易，并努力用这些属性来制作软件。
  • 修改代码需要勇气和技巧。不是鲁莽。
  • 通常，最好是进行一系列频繁的、小的、可验证的调整，而不是一次大范围的代码更改。
  • 避免复制粘贴编码。将您的逻辑分解到共享函数和公共库中，而不是遭受重复的代码(和重复的错误)。
  • 编写小的模块化代码段。保持干净整洁。
  • 代码应该是“共享的”,因为它对多个客户端有用，而不是因为开发人员想要创建一个漂亮的共享库。
  • 不要轻视别人的代码。使用现有的库可能比编写自己的版本更好。
  • 创建软件版本需要纪律和计划。这不仅仅是在开发人员的IDE中点击“构建”。
  • 总是从头开始，在一个全新的地方构建你的软件。永远不要重用软件构建的旧部分。
  • 让您的构建变得简单，只需一个步骤就可以自动完成整个过程。使用脚本语言来完成这项工作。
  • 在CI服务器上部署您的构建以确保它们的健康。从同一个系统进行正式发布。
  • 处于不断学习的状态。总是寻求学习新的东西。
  • 我们的学习往往过于狭隘。考虑更广的参考范围。从多个领域汲取灵感。
  • 学习时记笔记。即使你把它们扔掉。
  • 如果不能简单的解释，说明你理解的不够好。
  • 教一个题目，才能学好那个题目。
  • 我听见了，我忘记了。我看见了，我记得。我知道，我也理解。
  • 用你刚刚学到的东西巩固你的记忆。尝试例子，回答问题，创建宠物项目。
  • 警惕停滞。寻求成为更好的程序员，顾名思义，并不是最舒服的生活方式。
  • 期望投入时间和精力来提高你的技能。这是一项值得的投资；它会回报自己的。
  • 不要通过编写不可读或不必要的“聪明”代码来使自己变得“不可或缺”。
  • 尊重软件许可证。
  • 不要变成一招小马。作为一名开发人员，让自己面对新的挑战、学习和成长。
  • 热爱所有语言
  • 学习遵循不同习惯用法和范例的语言。
  • 考虑学习一些不再常用的“死”语言，以了解你的工艺历史。
  • 一个好的程序员知道多种语言和多种习惯用法，拓宽了他们的解决方案。这改进了他们编写的代码。
  • 使用你的编程语言是你每天都要处理的一种关系。
  • 为了用一种语言写出最好的代码，你应该遵循它的风格和习惯用法，而不是把你自己的风格和习惯用法强加给它。
  • 好的程序员是好的沟通者。他们说、写、编码、听、读都很好。
  • 不要期望一夜之间成为语言大师，也不要在努力的过程中感到沮丧。
  • 首先把精力集中在最重要的事情上。什么是最紧急的，或者将产生最大的价值？
  • 如果你经常做某事，让计算机为你做。用脚本实现自动化。
  • 把大任务分成一系列更小的、更容易理解的任务。通过这些你将能够更准确地判断进步。
  • 确保你定义了“完成”
  • 如果你不知道什么时候完成，那么你就不应该开始。
  • 使用用代码编写的测试来定义您的代码何时完成并工作。
  • 不要做不必要的工作。工作直到你“完成”然后停下来。
  • 当面对一个问题时，确保你已经考虑了不止一种方法来解决它。只有到那时，你才应该开始努力。
  • 有目的地把自己放在优秀的程序员旁边。
  • 更多的注释不一定会让你的代码更好。交流代码不需要额外的注释来支撑。
  • 了解开发方法、趋势、宣言和时尚。
