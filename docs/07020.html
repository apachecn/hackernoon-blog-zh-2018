分散待办事项列表应用程序的优势和潜在使用案例

原文：https://medium.com/hackernoon/how-a-to-do-list-application-when-decentralized-can-potentially-solve-3-problems-c7757579ad62?source=collection_archive---------34-----------------------#2018-08-20

我们使用过像“Google Keep”或“Evernote”这样的应用程序来记笔记和列出待办事项。根据我的观察，当我们在区块链平台上部署它的分散版本时，这个简单的应用程序有3个用例。我现在将解释应用程序设计及其3个用例。待办事项列表应用程序的这些用途更加安全、私密和防篡改。

应用程序设计:

像任何待办事项应用程序一样，我们有用户帐户，供用户在其中创建笔记。该应用程序需要使用solidity或任何其他面向合同的图灵完整语言编写。一旦开发了应用程序，就需要将其部署在区块链上。

让我总结一下参数:

  1. 用户帐户。
  2. 用户书写的笔记列表。
  3. 用户的地址。这是写笔记的用户的公共地址。在区块链上，我们将通过此参数向用户推荐。
  4. 时间戳，即我们部署合同的时间。这对区块链的生态系统非常重要。

现在让我们来看一下这个应用的三个用例。

用例1:分散的Twitter

待办事项应用可以成为像Twitter这样的微博平台的基础。

为了帮助您理解这一点，让我来推导一下我们的待办事项列表应用程序和这个平台之间的类似关系:

  1. 待办事项列表中的用户账户将成为这个平台上的用户档案。
  2. 笔记列表将是用户的推文。
  3. 地址将是用户用来访问其账户的地址。
  4. 时间戳是用户发推文的时间。

这样我们就可以拥有一个像Twitter一样的分散式微博平台。

用例2:计费系统

分散式计费系统可以按如下方式实现:

  1. 待办事项列表中的用户账户将是为每个客户创建的单独账户。
  2. 票据清单是与特定客户相关的票据。
  3. 顾客可以使用地址查看他们以前的购买情况。
  4. 时间戳是为客户生成账单的时间。

优点:由于法案是在区块链，这将有利于防篡改审计跟踪管理。

用例3:过程管理系统

每个公司都将他们的流程保存在文档中。现在，这种方法的问题是，有时进程可能会丢失或被黑客攻击或意外删除。因此，需要一个分散的过程管理系统。

它可以按如下方式实现:

  1. 待办事项列表中的用户账户将成为一个单独的进程。
  2. 注释清单是流程中每一步的程序或指令。
  3. 一个地址可以用来读/写一个特定的进程。出于安全原因，只有信任的人才能访问数据。
  4. 时间戳是修改流程时的时间。

一旦这个系统在私人区块链上，它就不能被删除或入侵。

这是分散式待办事项列表应用程序的三个用例。
