# 如何调查和解决生产问题？

> 原文：<https://medium.com/hackernoon/how-to-investigate-fix-production-issues-92375e095578>

## 如果您在生产环境中发现一些重复出现的问题，您会怎么做？

![](img/7409ecc858ec5748d27392fc0f7e43d6.png)

令人惊讶的是，重启在大多数情况下都是有效的，但是如果您在多个数据中心管理 10 项服务和 100 台具有代码的机器，那么即使重启也可能是一项全职的维护工作。我见过管理微服务的团队不得不保留 2-3 名软件工程师来管理服务并不时重启服务器。

并且[生产](https://hackernoon.com/tagged/production)问题绝不罕见，维护有许多未知生产问题的服务可能会非常非常痛苦，这可能会导致各种各样的问题。

1.  人们开始寻找新的工作，这会产生一个反馈循环，使问题变得更糟。
2.  开发人员经常分心去解决问题
3.  从事实际的新任务变得很困难
4.  随着时间的推移，人们和团队可能会害怕发布新的特性
5.  经理们不再信任他们的团队成员，开始想出许多流程来阻止人们发布东西

好了，现在我们已经确定这是一个严重的问题。有什么解决办法？

基本上，我认为解决生产问题有两个问题

1.  优化
2.  技术挑战/调试

我已经看到，有时候接下修复[问题](https://hackernoon.com/tagged/issue)的任务可能比实际解决问题更大，我认为这有多种原因。

1.  生产问题不知从哪里冒出来，它与计划任务搞砸了，所以我们倾向于忽略它。
2.  很难预测解决这个问题需要多长时间

3.**“就发生了几次”，**我们莫名其妙地希望它自己消失。

4.对许多人来说，一次又一次地重启似乎是一个更容易的选择。

解决上述问题可能很棘手。你必须说服你的经理和团队成员停止某项工作，继续调查。但是让我们假设你以某种方式得到了它的优先级，那么你如何解决它呢？

随着时间的推移，我了解到调试或任何生产调查就像一个解决问题的科学过程。有几件事你必须记住。

1.  在对这个问题做出任何假设之前，试着去了解这个问题周围的系统。围绕它做很多实验。如果需要，绘制图表，做笔记等。**这一点至关重要，因为如果你在不了解系统的情况下对问题做出一个或多个假设。很难从这些假设中走出来，看到全貌。**
2.  尽量在最小设置下重现问题，这样可以更快地进行实验和尝试。最终，这也将帮助您验证修复。
3.  **先验证简单假设**。很多时候，我们会提出一些复杂的假设，这些假设也很难得到验证，但问题最终会变得非常简单。
4.  很多时候，这个问题是某个库中的已知问题，在 StackOverflow 上搜索，或者在 StackOverflow 上发布一个新问题，提供所有详细信息。**你会惊讶于仅仅通过张贴问题就能获得的洞察力，因为这迫使你在一个最小重现场景中构建问题**。场外 StackOverflow 社区也很棒，所以他们可以帮你解答。
5.  如果你仍然找不到根本原因，把问题交给你的同事，让他从一个新的角度来看问题。有时候很难从自己的假设中走出来。
6.  如果你仍然找不到问题，那么记录你的主要发现，找到一个解决方法(这应该比每次问题发生时重新开始要好:P)然后继续前进。
7.  从错误中学习，下次会更好。

我看到很多很多的时间被浪费在解决问题上，所以如果这能帮助更少的人更快地解决问题，那么写这篇文章是值得的。如果你喜欢这个内容，请鼓掌(👏)并让我知道你的想法？
