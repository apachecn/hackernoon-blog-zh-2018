# 用讲故事的技巧写一篇科技报告

> 原文：<https://medium.com/hackernoon/writing-a-tech-talk-using-storytelling-techniques-ff116392b4d8>

做技术演讲和讲故事没什么不同。

![](img/2d317ba9c43e79a729a7e377c9384c05.png)

有些读过我博客的人可能已经知道[讲故事](https://hackernoon.com/software-and-the-storyteller-f671fb3b1876)非常贴近我的心。对我来说，生活就是一系列的故事，从我们与其他人的互动，到我写的软件。最近有人问我如何写技术报告。当我思考这个问题时，我意识到我是像写故事一样对待它的。我从来没有列出我遵循的步骤(老实说，我不会以任何结构化的方式来做)，但一旦它在我脑海中出现，我认为它可能值得分享，以防它对其他人有所帮助。

![](img/96ffe3fd1d6ce11c48f405bf00fc814e.png)

## 写下你的目的

重要的事情先来。你需要知道你谈话的目的。这是最关键的一块。如果你不知道你为什么要讲这个故事，那么其他任何事情都没有意义。在此之后的一切都必须服务于这个目的。

> 我为什么要讲这个故事？
> 我想让人们知道/学到什么？

这是值得花一点时间的。我知道有时候我开始认为我的目标是一件事，但在某个晚上的凌晨 3 点，我意识到根本不是。其他时候，我从一开始就非常清楚我的目标是什么。一旦你知道*为什么*你要讲这个故事，那么这个故事是什么就会水落石出。

![](img/0ab1f24bafee3747ad3b4cff3fd2d0ac.png)

## 完成你的故事

每个故事都遵循一个非常基本的模式:

> 问题是什么
> 你做了什么或没做什么来解决它
> 结果是什么

不要纠结于“问题”这个词。“问题”可以是您需要解决一个业务问题，因此您正在讲述您构建的应用程序的故事。但可能是你开始使用一项新技术，现在在分享如何入门的教程。

你讲的故事需要有重点。做到这一点的最好方法是首先确定你想在哪里结束。然后在此基础上，计算出你可以从多接近的距离开始**这样故事仍然有意义。这可能看起来是错误的，但它会阻止你从计算机刚发明的时候开始，永远不会真正完成任何事情。**

你可以在场景设置中给出一些背景故事，但故事的核心应该丰富而详细，所以你要确保你没有试图融入太多！你将不得不遗漏一些东西——特别指出一些你没有涵盖的东西也是可以的。如果你发现这一点很难，回头看看你的目标，让它来引导你！

![](img/d6a6f096d4d1e1295c9a56a85c3d8eaf.png)

## 谁是我的受众，我可以做出哪些假设？

通过演讲，你应该能够对你的听众有所了解。值得思考的是，故事的哪一部分对他们最重要——回顾你的故事情节，它是否涵盖了他们想知道的内容？

> 他们关心什么？他们期望学到什么？
> 对于他们已经知道的东西，我可以做出什么样的假设？

能够对观众会知道什么做出假设是很重要的。你不可能面面俱到，但是你确实需要确保你的听众了解足够多的信息来跟上你的故事。在 javascript 会议上，你应该假设人们已经知道 Javascript 是什么，但是你也可以假设他们知道 IDE 是什么，或者命令行是什么。如果你给学校的孩子们做报告，没有任何理由认为他们会知道这些。

![](img/7f54c79e71a1eb11759a866a6a46e814.png)

## 有哪些人物？

我发现把所有的技术、应用和人都当作我故事中的角色非常有用。就像一个作者会做的那样，我为自己做一些笔记来帮助我塑造我的故事。这个片段是给你的**，你应该慷慨地给自己提供信息，但只给观众他们需要的故事和*不多也不少。***

**每个字符应该具有以下内容:**

> **名称
> 一句话描述
> 背景故事**

**你会发现，你每天都在处理很多事情，如果让你描述，你会很难简明地解释，确切的原因是你太熟悉了，以至于你从来没有想过。确保你可以用一句话描述你正在谈论的任何事情(不管你是否真的在谈话中分享了这些)，这可以让你的想法更加清晰。**

**背景故事也很重要。再次，*这是给你的*。你将根据它的重要性、听众以及最重要的你的目的来决定分享哪些内容。想写多少就写多少——你用的是什么版本的库，有没有什么问题，它替换了什么。这有助于你思考它是如何融入故事的。**

**有些人物在谈话中永远不会提到，其他的只是一带而过。一些将在场景设置中出现，以帮助观众理解故事，另一些将随着故事的进展被有机地引入。**

**记住**你**是这个故事里的一个人物！人们也会想了解你一点。确保你也给自己做了介绍。**

**![](img/663d4c7c28e56cdfb135eb2537d66d28.png)**

## **作事前的现场（或情况）描述**

**你应该尽可能集中核心故事，但是很可能你将不得不根据观众做一些场景设置。回到你的假设和你的角色描述，找出在你的故事有意义之前需要理解什么。**

**不要给观众提供超过他们需要的信息。他们想听好的东西——尽可能快地抓住核心故事。**

**![](img/77b4c1123df3865a2cc7e5bf86074c0a.png)**

## **无情地编辑**

**没有一个初稿是完美的。有时甚至第 50 稿也没有通过。如果你发现事情不像你想象的那样，不要害怕完全返工。不要害怕扔掉它，重新开始。**

**请注意，如果你没有从你的初稿中删掉一些东西，那么你可能不够无情。大声练习——你的时间安排如何，也许跑得有点长？回到目的。剪掉所有不需要的东西——这会很痛。也许这些片段会出现在另一个故事中。**

**![](img/d1cfc1c5281e1aaf415d730a1eee1aff.png)**

## **添加一个后记**

**所有的谈话都需要有一个尾声。它应该提醒人们你是谁，以及他们如何与你联系——甚至可能是续集的细节！确保你分享的任何联系方式都不会让任何人知道。**

**![](img/7fec8edd70f57b2dceaa56ddbfe9298c.png)**

## **做你自己！**

**这些都不是规则，没有完美的方法来准备或发表演讲。做你自己，你会很棒的。**
